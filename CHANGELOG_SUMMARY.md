# 最近の修正内容サマリー

このセッションで解決された主な問題と改善点の概要です。

______________________________________________________________________

## 1. TerraformとSQLによるテーブル作成の役割分担を明確化

- **問題:** `README.md`ではTerraformが全てのテーブルを作成するような誤解を招く記述になっていた。
- **解決策:** Terraformが作成するテーブルと、手動でのSQL実行が必要なテーブル（申請履歴、帳票用テーブル等）を`README.md`に明確に分けて記載した。

## 2. 履歴テーブルの命名と役割を整理

- **問題:** ドキュメント内に存在しないテーブル名 (`iam_policy_permissions_history`) が記載されており、「生の履歴」と「帳票用の整形済み履歴」の役割が曖昧だった。
- **解決策:** ドキュメントを修正し、`iam_policy_bindings_raw_history`（生の履歴）と`iam_permission_bindings_history`（帳票用）の役割を明確にした。

## 3. デプロイ・運用関連の改善

- **問題:** `bootstrap-deploy.sh`がSQLを誤った順序（アルファベット順）で実行し、デプロイに失敗する可能性があった。

- **解決策:** `bootstrap-deploy.sh`を修正し、テーブルとビューの依存関係を考慮した正しい順序でSQLを実行するようにした。

- **問題:** `terraform output`で確認できるCloud Schedulerジョブ名が一部不足していた。

- **解決策:** `terraform/outputs.tf`に不足していた3つのジョブ（不整合検知、期限切れ権限剥奪、履歴更新）の出力定義を追加した。

## 4. セキュリティと拡張性の向上 (Terraformリファクタリング)

- **問題:** 実行サービスアカウントに過剰な`organizationAdmin`権限が付与されており、最小権限の原則に反していた。

- **解決策:** `terraform/main.tf`から`organizationAdmin`ロールの付与を削除した。

- **問題:** IAMロールの定義が分散しており、将来の機能追加時に変更が煩雑になる可能性があった。

- **解決策:** `terraform/main.tf`をリファクタリングし、`for_each`を使用してロールのリストから動的にIAM権限を付与する構造に変更。これにより、ロールの追加・削除がリストの編集だけで済むようになった。この設計方針を`README.md`にも追記した。

## 5. ドキュメントの明確化

- **問題:** Webhook認証に必要なSecret Managerのシークレットについて、事前作成の必要性や設定方法が不明確だった。
- **解決策:** `README.md`に、`WEBHOOK_SECRET_NAME`等の具体的な変数名と、`saas.env`での設定方法を追記した。

## 6. クリティカルバグの修正

- **問題:** 期限切れ権限を剥奪するCloud Runジョブ (`/revoke_expired_permissions`) が、不変オブジェクトの属性を変更しようとしてクラッシュしていた。
- **解決策:** `cloud-run/app/main.py`を修正し、`dataclasses.replace()`を用いて新しいオブジェクトを生成することで、エラーを解消した。

## 7. セキュリティの改善 (Cloud Run)

- **問題 (セキュリティ):** Webhook認証のトークン比較に`==`演算子を使用しており、処理時間の差からシークレットを推測されるタイミング攻撃に対して脆弱だった。
- **解決策 (セキュリティ):** `secrets.compare_digest()`を使用するように変更し、処理時間を一定にすることで脆弱性を修正した。

## 8. 堅牢性の向上 (Cloud Run)

- **問題:** IAMポリシーの同時更新が発生した場合、APIが返す競合エラー（409 Conflict）をハンドリングできず、処理が失敗していた。
- **解決策:** `iam_executor.py`を修正し、競合エラー発生時に最新のポリシーを再取得して最大3回まで自動でリトライするロジックを実装。これにより、同時更新時にも処理が成功する可能性が高まった。

## 9. 開発・デプロイ環境の最適化

- **問題:** ローカル開発環境のPythonバージョンとCloud Run環境のPythonバージョンが異なっていた。また、`Dockerfile`がPoetryを使用しておらず、`requirements.txt`が存在しないためビルドできない状態だった。
- **解決策:**
  - `cloud-run/pyproject.toml`のPythonバージョン指定を`^3.12`に統一し、ローカル環境のPythonバージョンも3.12に切り替えた。
  - `cloud-run/Dockerfile`をPoetryを使ったマルチステージビルドに全面的に書き換え、本番用コンテナイメージのビルドプロセスを堅牢かつ効率的にした。

## 10. パフォーマンス改善 (Cloud Run)

- **問題:** 期限切れ権限剥奪ジョブで、ループ内で申請ごとにクエリが発行され、パフォーマンスが著しく低下する「N+1問題」が発生していた。
- **解決策:** `repository.py`を修正し、1回の`LEFT JOIN`クエリで必要な情報をまとめて取得するように変更。ループ内の都度のクエリ発行をなくし、パフォーマンスを大幅に改善した。また、この修正を検証するユニットテストも追加した。

## 11. 堅牢性の向上 (Google Apps Script)

- **問題:** Google Apps ScriptからCloud Runへの呼び出しが、一過性のエラーやCloud Runのコールドスタート時に失敗すると、リトライされずに処理が中断していた。
- **解決策:** `apps-script/Code.gs`の`callCloudRunExecute_`関数に、最大3回のリトライとエクスポネンシャル・バックオフ（1秒、2秒...と待機時間を増やす）のロジックを実装し、Cloud Runとの連携処理の堅牢性を向上させた。

## 12. セキュリティ強化: VPC-SCとCloud RunのIngress制限を構成可能に

- **問題:** Cloud Runのエンドポイントがインターネットに公開されており、第三者からのアクセスが可能だった。
- **解決策:** Terraformに変数を追加し、`enable_vpc_sc`フラグを`true`にすることで以下のセキュリティ設定が自動的に有効になるようにした。
  - **Cloud Run Ingress**: `INGRESS_TRAFFIC_INTERNAL_AND_CLOUD_LOAD_BALANCING`に変更され、内部トラフィックとロードバランサ経由に限定される。
  - **VPC Service Controls**: 関連するGoogle APIへのアクセスを制限するサービス境界（Perimeter）が構築される。
  - **APIの有効化**: VPC-SCに必要な`accesscontextmanager.googleapis.com` APIがフラグに連動して有効化される。
- **効果**: これにより、本番環境向けの高度なセキュリティ要件に、Terraformの変数を変更するだけで柔軟に対応できるようになった。

## 13. クリティカルバグの修正 (Cloud Run ジョブ)

- **問題:** Cloud Runの定期ジョブ (`/reconcile`, `/jobs/update-iam-bindings-history`) が、コンテナ内に存在しないSQLファイルを相対パスで読み込もうとして `FileNotFoundError` でクラッシュしていた。
- **解決策:** 外部のSQLファイルを読み込む設計を廃止し、SQLクエリを `repository.py` 内に直接埋め込むように変更。これにより、Dockerのビルドコンテキストに依存しない堅牢な設計となり、エラーを完全に解消した。また、この修正を検証するためのユニットテストも追加した。

## 14. 重大なセキュリティ脆弱性の修正 (認証バイパス)

- **問題:** Webhook認証において、環境変数 `WEBHOOK_SHARED_SECRET` が未設定の場合に認証がバイパスされ、誰でもエンドポイントを実行できてしまう「フェイルオープン」の脆弱性が存在した。
- **解決策:** `cloud-run/app/main.py` の `_authorize` 関数を修正し、シークレットが未設定の場合は常にアクセスを拒否する「フェイルクローズ」の設計に変更。これにより、設定ミスや環境変数の読み込みエラーが発生した場合でも、システムの安全性が保たれるようになった。

## 15. 設定の不整合を修正 (テーブル名の統一)

- **問題:** パイプラインジョブの結果を記録するテーブル名が、Terraform (`iam_pipeline_job_reports`) とアプリケーションコード (`pipeline_job_reports`) で異なっていた。これにより、Terraformで作成されたテーブルが使われず、アプリがエラーとなる不整合が発生していた。
- **解決策:** `cloud-run/app/repository.py` と `sql/001_tables.sql` のテーブル名をTerraformの定義に合わせて `iam_pipeline_job_reports` に統一し、インフラとアプリケーション間の設定の整合性を確保した。

## 16. 機能追加: 異常検知時のプロアクティブな通知 (アラート自動化)

- **問題:** バッチ処理の失敗や実行時エラーがBigQueryには記録されるものの、管理者が能動的に確認しに行かない限り異常を検知できない状態だった。
- **解決策:** GCPネイティブの仕組みを活用し、プロアクティブなアラート通知機能を実装した。
  - **Python:** アプリケーションのエラーハンドリング箇所で、標準の`logging`ライブラリを用いて`ERROR`レベルのログを出力するように修正。
  - **Terraform:**
    - `alert_notification_email` と `alert_notification_webhook_url` 変数を追加し、アラートの通知先（メール, Webhook）を設定可能にした。
    - Cloud Runサービスから`ERROR`レベル以上のログが検出された場合に発火する、ログベースのCloud Monitoringアラートポリシーを追加。
    - 設定された通知先（メール/Webhook）に応じて、Terraformが動的に通知チャネルを作成し、アラートポリシーに紐付ける構成とした。
- **効果:** これにより、アプリケーションでエラーが発生した際に、管理者が即座にメールやチャットで通知を受け取れるようになり、監視負荷の軽減と迅速な障害対応が可能になった。
