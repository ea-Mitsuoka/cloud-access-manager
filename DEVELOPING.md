# ローカル開発ガイド (Poetry)

このガイドでは、Poetry を使用してローカル開発環境をセットアップするための手順を説明します。これらの手順に従うことで、単体テストを実行し、変更をコミットする前に検証することができます。

## 1. 前提条件

始める前に、ローカルマシンに以下のツールがインストールされ、設定されていることを確認してください。

- **Poetry**: Python の依存関係管理およびパッケージングツール。
- **gcloud CLI**: Google Cloud のコマンドラインツール。
- **Terraform**: Infrastructure as Code ツール。
- **Python**: バージョン 3.12 (プロジェクトの `pyproject.toml` で指定)。
- **Docker**: コンテナイメージをビルドするため（ローカルテストではオプション）。

また、開発およびテスト用の Google Cloud プロジェクトへのアクセス権があり、`gcloud` CLI が認証済みである必要があります (`gcloud auth login`)。

## 2. 設定

1. **`saas.env` ファイルの作成**:
   リポジトリのルートに `saas.env` ファイルがない場合は、サンプルファイルをコピーして作成します。

   ```bash
   cp saas.env.example saas.env
   ```

1. **`saas.env` の編集**:
   `saas.env` を開き、ローカルセットアップに必要な値を入力します。主に以下の項目です。

   - `TOOL_PROJECT_ID`: 開発用の Google Cloud プロジェクトID。
   - `REGION`: リソースの GCP リージョン。
   - `BQ_DATASET_ID`: テスト用の BigQuery データセットID (例: `iam_access_mgmt_dev`)。

## 3. 環境のセットアップとテスト

Poetry は環境のセットアップとコマンドの実行を簡素化します。

1. **`cloud-run` ディレクトリへの移動**:
   Python プロジェクトはこのディレクトリで定義されています。

   ```bash
   cd cloud-run
   ```

1. **依存関係のインストール**:
   このコマンドは、ディレクトリ内に新しい仮想環境を作成し、`pyproject.toml` で指定されたすべてのアプリケーションおよび開発依存関係をインストールします。

   ```bash
   poetry install
   ```

1. **リンティングとテストの実行**:
   変更を検証するために、リンターとテストスイートを実行します。

   - **コードスタイルのチェック**:
     `flake8` を実行して、コードがスタイルガイドラインに準拠していることを確認します。

     ```bash
     poetry run flake8
     ```

   - **単体テストとシナリオテストの実行**:
     `pytest` を使用して、テストスイート全体を実行します。これには、個々の関数を検証する単体テストと、重要なワークフローを保証するシナリオテストが含まれます。

     ```bash
     poetry run pytest
     ```

     テストスイートには以下のものが含まれます:

     - **実行フローテスト**: 権限の付与と取り消しのAPIワークフローを検証します (`test_main_execute_flow.py`, `test_main_revoke_flow.py`)。
     - **コレクターフローテスト**: Google Groupとリソースインベントリのデータ収集パイプラインが正しく機能することを保証します (`test_collectors_flow.py`)。
     - **その他単体テスト**: 個別のコンポーネント（Executor, Job, Validatorなど）の振る舞いを検証します。

## 4. アプリケーションのローカル実行 (基本的なチェック)

基本的なチェックのために、Flask アプリケーションをローカルで実行できます。

**重要な注意**: ほとんどのエンドポイントは、実際の Google Cloud 認証と権限を必要とします。ローカルでの実行は、エンドツーエンドのテストのために開発環境にデプロイする代わりには**なりません**。

1. **`cloud-run` ディレクトリへの移動** (まだ移動していない場合)。

1. **環境変数の設定**:
   アプリケーションは `saas.env` の環境変数に依存しています。まず、**リポジトリのルート**から同期スクリプトを実行して、`.env` ファイルが最新であることを確認します。

   ```bash
   # From the repository root
   bash scripts/sync-config.sh
   ```

   Poetry は `pyproject.toml` と同じディレクトリに `.env` ファイルが存在する場合、自動的に変数を読み込むことができます。`sync-config.sh` スクリプトはすでに `cloud-run/.env` を作成しているため、手動での `export` は不要です。

1. **Flask アプリの実行**:
   `poetry run` を使用して、管理された環境内で Flask を実行します。

   ```bash
   # From the ./cloud-run directory
   poetry run flask --app app/main run
   ```

1. **ヘルスチェックのテスト**:
   新しいターミナルで、ヘルスチェックエンドポイントにアクセスできます。

   ```bash
   curl http://127.0.0.1:5000/healthz
   ```

   `{"ok":true}` というレスポンスが表示されるはずです。

## 5. ドキュメントの歩き方

機能追加やリポジトリ管理を円滑に進めるため、各ドキュメントの役割とユースケースを理解しておくことが重要です。

| ドキュメントファイル | 役割と目的 | 主なユースケース |
|---|---|---|
| **`README.md`** | **プロジェクトの顔**。システム全体の目的、アーキテクチャ、主要機能を概観します。 | 新規参画者（開発者、PM、監査担当）がプロジェクトの全体像を素早く理解するために使います。大規模な機能追加やアーキテクチャ変更があった場合は、このファイルを更新します。 |
| **`DEVELOPING.md`** | **開発者のための主要ガイド**。ローカル開発環境のセットアップ、テスト実行、コーディング規約などを提供します。 | 開発者が作業を開始する際の最初の参照先です。テスト手順の変更や、新しい開発ツールを導入した際に更新します。この「ドキュメントの歩き方」のようなメタ情報もここに含みます。 |
| **`docs/operations-runbook.md`** | **運用担当者のための手引書**。本番環境のデプロイ、管理、トラブルシューティング手順を詳述します。 | SREやインフラ担当が、システムの安定運用のために参照します。デプロイ手順やTerraformの適用方法、緊急時の対応フローなどを変更した場合は、必ずこのファイルを更新します。 |
| **`docs/untested-items-handover.md`** | **リスク管理台帳**。実装済みだが未検証の項目を一覧化し、関係者間で認識を合わせるために使います。 | 機能実装後に十分なテストができない場合、ここに項目を追加してリスクを明示化します。自動テストを追加して未検証項目をカバーできるようになった場合は、該当項目のステータスを「完了」に更新します。 |
| **`CHANGELOG_SUMMARY.md`** | **高レベルな変更履歴**。リリース間の主要な変更点を、ビジネス寄りの視点で要約します。 | 機能開発が完了・マージされた後、変更点の要約を一行で追記します。 |
| **各コンポーネントの `README.md`** | **特定領域の深掘りガイド**。`cloud-run/` や `sql/` など、特定コンポーネントの内部設計や詳細な仕様を説明します。 | Cloud RunにAPIエンドポイントを追加した場合など、特定コンポーネントの仕様変更があった際に、該当するREADMEを更新します。 |

## 6. ドキュメントサイト (MkDocs) のローカルプレビュー

本プロジェクトでは、統合ドキュメントサイトおよびPython APIリファレンスの生成に [MkDocs](https://www.mkdocs.org/) と [Material for MkDocs](https://squidfunk.github.io/mkdocs-material/) を使用しています。
ドキュメント（`docs/` 配下のMarkdownファイルやPythonのDocstring）を編集した際は、以下の手順でローカルサーバーを立ち上げ、表示をプレビューすることができます。

1. **`cloud-run` ディレクトリへの移動**:
   MkDocsと関連プラグインはPoetryの `dev` 依存関係としてインストールされているため、`cloud-run` ディレクトリ内でコマンドを実行します。

   ```bash
   cd cloud-run
   ```

1. **ローカルサーバーの起動**:
   リポジトリルートにある `mkdocs.yml` を指定してプレビューサーバーを起動します。

   ```bash
   poetry run mkdocs serve -f ../mkdocs.yml
   ```
