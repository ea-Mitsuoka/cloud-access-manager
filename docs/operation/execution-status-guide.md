# 申請ステータスと実行結果の読み解き方

Cloud Access Managerの管理UI（スプレッドシートの `requests_review` シートなど）には、「実行結果」「最終反映確認」「最終確認時刻」という3つの重要な列が存在します。

これらは、システムがAPIを叩いた\*\*「行動の記録」**と、GCPの現実を読み取った**「結果の観測」\*\*という2つの異なる視点を掛け合わせたものです。この組み合わせを読み解くことで、申請が今どういう状態にあるのか、あるいは何が原因でトラブっているのかを正確に把握することができます。

## 1. 各列の役割と更新タイミング

| 列名 | 意味 | 主な値 | 更新タイミング |
| :--- | :--- | :--- | :--- |
| **実行結果** | Cloud RunがGCPのIAM APIを叩いた際の「直接的な成否」 | `未実行`, `SUCCESS`, `SKIPPED`, `FAILED`, 実際のエラーメッセージ | 承認一括送信時、または「最新ステータス取得」実行時 |
| **最終反映確認** | 「実際のGCP環境に権限が存在するか」を現況と突き合わせた結果 | `未確認`, `反映済み`, `未反映`, `エラー` | 「最新ステータス取得」実行時（毎朝の収集バッチ完了後） |
| **最終確認時刻** | 上記2つのステータスをシステムが最後にチェックしたタイムスタンプ | 日時文字列 | 一括送信時、または「最新ステータス取得」実行時 |

## 2. ステータスの組み合わせパターン

運用担当者（SRE）の目線で、発生しうるすべての組み合わせパターンを整理します。

### 🟢 正常に進行しているケース（理想的な状態）

#### `未実行` × `未確認`

- **どういう状態？:** 申請が作成されたばかりで、まだ承認されていない（または「🔄 レビュー結果を一括送信」が実行されていない）状態です。
- **システムの解釈:** 実行ログが存在しないため、GCPの現状との突き合わせも行われません。
- **運用者の対応:** 内容をレビューし、承認作業を進めてください。

#### `SUCCESS` × `未反映`

- **どういう状態？:** Cloud RunがGCPのIAM APIを叩いて**付与（または剥奪）に成功した直後**の状態です。
- **システムの解釈:** APIは成功（200 OK）を返しましたが、毎朝走る「IAMポリシー収集バッチ（GCPの現況収集）」がまだ実行されていないため、「本当にGCP上に存在するか」の裏付けをとれていない状態です。
- **運用者の対応:** 正常です。翌朝のバッチが走れば、自動的に「反映済み」に変わります。

#### `SUCCESS` × `反映済み`

- **どういう状態？:** **100%完璧な成功状態**です。
- **システムの解釈:** システムによるAPI実行が成功しており、さらに現況収集バッチによって「間違いなくGCP上にその権限が存在する（または剥奪されている）」ことが裏付けられました。監査上も完璧な状態です。

### 🟡 システムが処理をスキップしたケース（冪等性の発動）

#### `SKIPPED` × `反映済み`

- **どういう状態？:** システムが処理を行おうとしましたが、\*\*「すでにGCPの現実が、申請の希望通りになっている」\*\*ため、無駄なAPI実行をスキップした状態です。
- **システムの解釈:** 「すでに権限を持っている人に新規付与しようとした」場合や、逆に「すでに権限がない人から剥奪しようとした」場合に発動します。また、過去に一度 `SUCCESS` している申請を再実行した場合もこれになります。
- **運用者の対応:** 現実の権限は正しい状態になっているため、何も問題ありません。

#### `SKIPPED` × `未反映`

- **どういう状態？:** システムは処理をスキップしたのに、GCPの現実には権限が反映されていないという**少し特殊な状態**です。
- **原因の例:**
  - ステータスが `APPROVED`（承認済）以外なのに実行APIが呼ばれたため、「承認されていないからスキップした」ケース。
  - 過去に一度 `SUCCESS` しているためスキップしたが、その後、誰かがGCPコンソールから手動でその権限を消してしまった（現実と申請の乖離が起きている）ケース。
- **運用者の対応:** 承認ステータスが正しいか確認するか、必要に応じてGCPの現状を確認してください。

### 🔴 エラー・異常系のケース（要トラブルシューティング）

#### `FAILED` (または詳細エラー) × `未反映`

- **どういう状態？:** **典型的なエラー状態**です。APIの実行に失敗し、GCPにも反映されていません。
- **原因の例:**
  - SaaS側のサービスアカウントにGCPのIAM付与権限（`Project IAM 管理者` など）が足りていない（403エラー）。
  - 対象のプロジェクトやリソースが存在しない（404エラー）。
  - 管理対象外のリソース（スコープ外）を指定したためにシステムがブロックした（`OUT_OF_SCOPE`）。
- **運用者の対応:** エラーメッセージを読んで原因を解消し、シートの「未反映の承認済リクエストを再実行」メニューからリカバリ（再実行）を行ってください。

#### `FAILED` (または詳細エラー) × `反映済み`

- **どういう状態？:** システムは「失敗した」と記録しているのに、GCPの現実を見ると「なぜか希望通りの権限がついている」という**矛盾した状態**です。
- **原因の例:**
  - **ネットワークエラー:** Cloud RunがAPIを叩いてGCP側で権限付与には成功した直後、ネットワークの瞬断などで「成功レスポンス」を受け取れず、Cloud Run側がタイムアウトで「失敗」と誤認したケース。
  - **人間の介入:** システムがエラーで失敗したのを見た運用者が焦って、GCPコンソール画面から手動で権限を付与してしまったケース。
- **運用者の対応:** 現実の権限としては要件を満たしていますが、システム上はエラー記録が残っているため、原因の特定と記録の整理が必要です（手動での介入は監査上好ましくありません）。
