# 未テスト項目 申し送り（テスト計画書）

最終更新日: 2026-04-06

この文書は、実装済みだが未検証の項目を残し、開発者とサポート担当（SRE/QA）が同じ前提で引き継ぐためのレジスタです。
システムが「フォーム＋GAS」から「統合SaaSポータル＋セキュアバックエンド」へ進化した最新アーキテクチャの要件を反映しています。

## 1. 運用ルール

- 未テスト項目が発生したら、このファイルへ必ず追記する。
- 検証完了したら `状態` を `完了` に更新し、実行ログ（証跡）を残す。
- 仕様変更で再検証が必要になったら `状態` を `再検証待ち` に戻す。
- 口頭連絡だけで終わらせず、必ずこのファイルに反映する。

状態の定義:

- `未着手`: まだ検証していない
- `進行中`: 検証中
- `完了`: 検証完了
- `再検証待ち`: 仕様/コード変更により再確認が必要

## 3. 変更時に追加すべき「類似の未テスト項目」

以下に当てはまる変更を入れた場合は、同じ形式で ID を追加してください。

- 外部連携の追加/変更 (Google APIs, Cloud Run API, Apps Script Webアプリ, Webhook)
- 認可・権限の変更 (Terraform IAMロール, SA切替, バックエンド認可ロジック)
- データスキーマの変更 (BigQueryテーブル/ビュー, 履歴列の追加)
- 定期実行の変更 (Cloud Scheduler, トリガー, バッチSQL)
- 承認フロー変更 (ステータス遷移、承認条件、再実行条件)

## 4. 証跡の残し方（推奨）

- BigQuery: 実行したクエリと取得結果（件数、代表行）
- Apps Script (SaaSポータル): Webアプリ画面のスクショ、またはGAS実行ログ
- Cloud Run/Scheduler: 実行時刻、HTTPステータス、エラーメッセージ

この証跡を運用チケットに貼り、ID（例: `UT-001`）を紐づける。

## 5. 実施チェックリスト（抜粋）

*(各UTを実施する際、このチェックリストをコピーしてチケット等に貼り付けて利用してください)*

### 5.1 UT-001 (Core Flow)

- [ ] Webアプリ（SaaSポータル）にアクセスし、フォームが正しく表示されること
- [ ] 必須項目を入力して送信し、エラーにならず BigQuery `iam_access_requests` に登録されること
- [ ] スプレッドシート `requests_review` に行が追加され、複数行を選択して「承認済」にコピペ変更できること
- [ ] 変更後、Cloud Runの `/api/requests/bulk-status` および `/execute` が走り、GCPに権限が付与されること

### 5.2 UT-003 (Security: 認可制御)

- [ ] 申請データAの「承認者メール」を `boss@example.com` として起票する
- [ ] スプレッドシートを `hacker@example.com` で開き、「承認済」に変更する
- [ ] Cloud Runのログに `Unauthorized status update attempt` が出力され、権限が付与されないこと（スキップされること）

### 5.3 UT-004 (Security: スコープ防御)

- [ ] 対象リソースに `folders/12345` と入力して申請する
- [ ] 承認して実行した際、APIが `OUT_OF_SCOPE` エラーを返し、システムがフォルダIAMを変更しないこと

### 5.4 UT-009 (Break-glass)

- [ ] Webアプリで申請種別「緊急付与」を選択して送信する
- [ ] スプレッドシートでの承認を待たずに即時権限が付与されること
- [ ] 設定されたWebhook/Emailに `[BREAK-GLASS]` アラートが即座に通知されること

### 5.5 UT-010 (UX: 却下通知と再申請)

- [ ] スプレッドシートで申請を「却下」に変更する
- [ ] 申請者宛に却下理由を添えたメールが届くこと
- [ ] メール内のリンクをクリックすると、前回の入力内容（リソースやロール名）が埋まった状態でWebアプリが開くこと
