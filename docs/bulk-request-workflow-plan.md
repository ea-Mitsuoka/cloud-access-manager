# IAM複数ロール一括申請およびBulk Update実行に関する仕様変更計画書

## 1. 背景と目的

現在のシステムでは、1つのリソース・ロールに対する申請と審査が1:1で紐付いており、複数の権限を同時に申請・審査する際のUX（ユーザー体験）が低下している。また、GASのセルの編集イベント（`onEdit`）をトリガーとした都度API実行は、将来的なシステム負荷やリミット超過（APIのレート制限等）のリスクを抱えている。
本仕様変更では、申請者が複数ロールを一括申請できる「カート方式」を導入し、承認者はスプレッドシート上で明細単位の審査を行った後、「Bulk Update（一括更新）」によって安全かつ効率的に処理を実行できるアーキテクチャへ移行する。

## 2. コアとなるデータモデルの変更

複数の申請明細を束ねる概念として、新たに **`request_group_id`（申請グループID）** を導入する。

- **データ構造の考え方**:
  - `request_group_id`: 1回の「申請アクション」に対して1つ発行されるUUID。申請理由や申請者情報はこれに紐付く。
  - `request_id` (既存): 申請グループの中に含まれる、個別の「リソース×ロール」の組み合わせごとに発行される明細行のUUID。

## 3. コンポーネント別の改修要件

### 3.1. BigQuery & Terraform (データベース・IaC)

既存のトランザクションテーブルおよびビューにカラムを追加する。

- **対象テーブル**: `iam_access_requests`, `iam_access_request_history`
- **追加カラム**: `request_group_id` (STRING, NULLABLE)
- **既存データへの対応**: 過去のデータに対しては、`request_group_id` に既存の `request_id` をそのままコピーする（1申請＝1グループとして扱う）データパッチ用のSQLを実行し、後方互換性を保つ。
- **ビューの改修**: GASが読み込む `v_sheet_requests_review` ビューにも同カラムを出力するようSQLを修正する。

### 3.2. SaaS Portal (フロントエンド・申請画面)

ユーザーが複数項目をまとめて申請できるUIへ変更する。

- **カートUIの導入**: リソースとロールを選択して「追加」ボタンを押し、画面上のリストに溜めていくUIを実装。
- **一括送信処理**: 「申請する」ボタン押下時、フロントエンド側で単一の `request_group_id` を生成し、各明細のペイロードに含めてバックエンドへ一括送信する。

### 3.3. Cloud Run (バックエンドAPI)

「1件ずつの処理」から「配列（リスト）を受け取ってのBulk処理」へエンドポイントの仕様を拡張する。

- **Bulk Update エンドポイントの作成** (`POST /api/v1/requests/bulk-review` 等):
  - リクエストボディとして、複数の `{ request_id, status, reject_reason }` を含むJSON配列を受け取る。
  - 内部でループ処理（または並列処理）を行い、GCPのIAM API呼び出しとBigQueryへの履歴書き込みを一括で行う。
  - **トランザクション管理**: 一部の権限付与が失敗した場合でも、成功したものはコミットし、レスポンスに「成功リスト」と「失敗リスト（エラー理由含む）」を明確に分けて返却する。

### 3.4. Google Apps Script (スプレッドシート・承認UI)

自動実行（イベント駆動）から、手動実行（バッチ処理）へUXを抜本的に変更する。

- **列の追加**: `requests_review` シートのA列付近に「申請グループID」列を追加（見やすさのため、同じIDが続く場合は条件付き書式で文字色を薄くする等の視覚的工夫を推奨）。
- **`onEdit` の無効化**: セルのステータスを変更した際の自動API実行ロジックを削除する（単なる下書き状態にする）。
- **カスタムメニューの実装**: メニューバーに「🔄 レビュー結果を一括送信」を追加する。
- **一括送信ロジックの実装**:
  - シート内の「ステータスが申請中以外」の行をすべて抽出する。
  - ステータスが「却下」の行が含まれる場合、`ui.prompt` を用いて一回だけ却下理由の入力を求める（複数の却下がある場合は共通の理由とするか、個別入力させるか要件定義）。
  - 抽出したデータをJSON配列に整形し、Cloud RunのBulk Updateエンドポイントへ送信。
  - 成功レスポンスを受け取った行だけを `requests_review` シートから削除し、`processed_history` シートへ移動（転記）させる。失敗した行はエラーダイアログと共にシートに残す。

## 4. リリース・移行手順 (Migration Strategy)

本変更はシステム全体に影響を与えるため、以下の順序で段階的にリリースを行う。

1. **データベース先行**: Terraformを適用し、BigQueryに `request_group_id` カラムを追加。既存データの互換性パッチ（UPDATE文）を実行する。
1. **バックエンド拡張**: Cloud RunのAPIをBulk Update対応版へデプロイ（既存の1件用APIは後方互換性のため残す）。
1. **フロントエンド＆GAS切替**: SaaS Portalをカート方式にデプロイし、同時にGAS側のコードをBulk Update方式に書き換える。
