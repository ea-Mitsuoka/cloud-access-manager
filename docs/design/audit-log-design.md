# アーキテクチャ決定記録 (ADR): 監査ログのLog Router Sink経由書き込みの不採用について

**最終更新日:** 2026-04-11
**対象コンポーネント:** Cloud Run (実行エンジン), BigQuery (監査・構成管理DB)

## 1. 背景

Cloud Access Managerの開発プロセスにおいて、システムとデータベースの疎結合化、およびエンタープライズ要件（既存SIEM連携など）への適応性を高める目的で、BigQueryへのデータ書き込みアーキテクチャの変更が検討された。

具体的には、現在の「Python（Cloud Run）からBigQuery SDKを用いて直接データ（申請や履歴）を書き込む方式」から、「Pythonは標準出力（`stdout`）へJSON構造化ログを吐き出すのみとし、Google Cloudネイティブの **Log Router Sink** を経由してBigQueryへストリーミングインサート（非同期書き込み）する方式」への移行改修に一度着手した。

しかし、検証の過程で本システムのワークフロー要件とBigQueryの仕様が致命的に衝突することが判明したため、改修をロールバックし、現在の直接書き込みアーキテクチャを維持する決定を下した。

## 2. 不採用となった3つの致命的な理由

本システムのBigQueryテーブルは、単なる「書き捨てのアクセスログ（Write-Only）」ではなく、申請・承認・付与・剥奪という状態（ステータス）の遷移を厳密に管理する **「構成管理DB（CMDB）およびステートマシン」** として機能している。
この特性において、Log Router Sink（非同期ロギング）の採用は以下の致命的なインシデントを引き起こす「時限爆弾」となることが判明した。

### 2.1 遅延によるステータス更新の空振り（緊急アクセスフローの破綻）

- **事象:** 緊急アクセス（Break-glass）フローでは、申請（`INSERT`）が送信された数ミリ秒後に、システムが自動でステータスを「承認済」に更新（`UPDATE`）して権限付与処理へ進む。
- **破綻理由:** Log Router Sinkを経由した場合、ログがBigQueryのテーブルに到達・反映されるまでに数秒〜数分の非同期遅延（レイテンシ）が発生する。そのため、直後に走る `UPDATE` 処理が「対象の申請ID（`request_id`）がまだ存在しない」と空振りしてしまい、緊急権限が永遠に付与されずにスタック（`PENDING`のまま放置）する重大障害が確定で発生する。

### 2.2 ストリーミングバッファ滞留中のDML制約エラー

- **事象:** Log Router Sinkは内部的にBigQueryのStreaming APIを使用する。BigQueryの仕様上、ストリーミング挿入されたデータは「ストリーミングバッファ（一時領域）」に最大90分間滞留する。
- **破綻理由:** バッファ内に滞留している行に対しては、`UPDATE` や `DELETE` などのDML（データ操作言語）を直接実行できない制約がある。そのため、ユーザーが申請してすぐにスプレッドシートで「承認」や「却下」を行おうとしても、DML制約に引っかかってエラーで弾かれるか、対象行がサイレントに無視されるという、ワークフローシステムとしての根本的な崩壊を招く。

### 2.3 状態と履歴の同期保証（一貫性）の喪失

- **事象:** 現在のシステム（`repository.py`）では、ステータスの更新と履歴（`iam_access_request_history`）の追記を1つのトランザクションのように同期的に処理している。
- **破綻理由:** 履歴だけを非同期ロギングに逃がした場合、「スプレッドシート（UI）上は『承認済』に変わっているのに、遅延のせいでBigQueryの履歴テーブルには承認の証跡がまだ存在しない」という一貫性のない（Eventually Consistentな）状態が生まれる。これは、監査の完全性を売りにするエンタープライズセキュリティ製品において、顧客の不信感を招くUXの悪化に直結する。

## 3. 結論：直接書き込み（DML / 同期処理）の維持と優位性

上記の理由から、「Log Router Sinkによる非同期書き込み」は本システムには不適合であると結論付けた。

現在の **「Pythonバックエンドから直接 `INSERT INTO` などのDMLを用いて、同期的に書き込みと更新を行う設計」** こそが、ステートマシンとしての堅牢性と監査ログの即時一貫性を両立する、本SaaSにおける\*\*絶対的な最適解（Best Practice）\*\*である。

## 4. 今後のマルチテナントSaaS化に向けた展望

将来的な「BYOC（顧客環境でのデプロイ）」や「マルチテナント型SaaS（顧客ごとにBigQueryプロジェクトを分離するフェーズ1）」を見据えた場合でも、この直接書き込みモデルは極めて高い適性を持つ。

Log Router Sink方式でマルチテナント化しようとすると、「ログのルーティングルール（FilterとSink）を顧客の数だけ複雑に管理・プロビジョニングし続けなければならない」というインフラの運用限界（スケーラビリティの壁）にぶつかる。
一方、現在の直接書き込みモデルであれば、Pythonバックエンドの初期化時（リクエスト受領時）に、HTTPヘッダーのテナント識別子に基づいて **「宛先となるBigQueryプロジェクトIDを動的に切り替える（ファクトリパターン）」** だけで済む。インフラを複雑化させることなく、現在の堅牢なコードベースを維持したまま、安全かつスピーディなマルチテナントデータ分離が実現可能である。
