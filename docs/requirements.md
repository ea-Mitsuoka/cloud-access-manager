# Cloud Access Manager 要件定義書

本ドキュメントでは、Cloud Access Managerの目的、対象範囲、機能要件、非機能要件、および将来の拡張方針について定義します。

## 1. 目的

- IAM付与申請から承認・実行・棚卸しまでを一気通貫で管理する。
- 最小構成は `Googleフォーム + Google Apps Script + Cloud Run + BigQuery` とする。
- インフラ構築は Terraform を必須とする。

## 2. 対象範囲

- 申請受付（Googleフォーム）
- 承認管理（スプレッドシート）
- IAM変更実行（Cloud Run経由のIAM API）
- 管理表生成（最新状態 + 申請/承認履歴）
- 定期棚卸し（現状の全量洗い出し）

## 3. 機能要件

### 3.1. 申請登録

- 必須項目未入力は受付不可とする。
- `request_id` は一意採番する。

### 3.2. 承認フロー

- 初期ステータスは `PENDING` とする。
- 承認者のみ `APPROVED/REJECTED` を更新可能とする。

### 3.3. IAM実行

- `APPROVED` かつ未実行のみ対象とする。
- 同一 `request_id` の有効実行は1回とする。
- 失敗時の再実行は許可し、履歴は追記する。

### 3.4. 定期棚卸し

- 日次または週次で全量収集を行う。
- 不一致を `iam_reconciliation_issues` に記録する。

## 4. 非機能要件

- 監査性: すべての状態変更に `who/when/what` を記録する。
- セキュリティ: 実行SAは最小権限、承認権限と実行権限を分離する。
- 可用性: 再試行設計、失敗時の再送手段を用意する。

## 5. 制約と注意点

- `WRITE_TRUNCATE`（最新用）と`WRITE_APPEND`（履歴用）を厳密に分離する。
- 緊急時の例外付与（Break-glass）フローを別途定義する。
- 期限付き権限の自動剥奪ジョブを用意する。
- **IAM Condition非採用の原則**: 期限付き権限の管理において、GCP標準のIAM Condition（条件付きバインディング）は使用せず、自前の夜間バッチによる能動的な剥奪（REVOKE）を採用する。
  - **理由1 (監査証跡の完全性)**: IAM Conditionによる期限切れは、GCPのCloud Audit Logsに「権限を剥奪した」という明示的なアクションイベントとして記録されないため、コンプライアンス上の証跡不備（いつ消えたか証明できない問題）に繋がる。
  - **理由2 (ステータス管理の統一)**: システム（BigQuery）上のステータスを確実かつ能動的に `APPROVED` から `REVOKED` へ遷移させることで、Single Source of Truth を保ち、UIとの不整合を防ぐ。
  - **理由3 (互換性の担保)**: IAM ConditionがサポートされていないレガシーなGCPリソースや、特定の組織ポリシー制約下での予期せぬ付与エラーを回避する。
- 認証はOIDCトークン認証をサポートする。

## 6. 将来拡張方針

- 申請UIをBacklogへ移行可能な構造にする。
- 承認済み申請をGitHub PRに変換し、レビュー統制と監査証跡を強化する。
- **VPC-SC対応のためのフロントエンド刷新**: AppSheet（Enterprise）やIAPで保護された内部Webアプリを申請フロントエンドとして導入する。
