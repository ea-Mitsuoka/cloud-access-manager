# IAM権限申請 ユーザーガイド

本ドキュメントは、Google Cloud環境のIAM権限（アクセス権）を申請するユーザー向けのマニュアルです。
本システムを利用することで、権限の申請から承認、実際のシステムへの反映までがセキュアかつ自動的に行われます。

______________________________________________________________________

## 1. 申請の基本フロー

権限の申請は、専用の**Googleフォーム**から行います。（※フォームのURLは管理者から共有されたものを使用してください）

申請後、指定した承認者（上長など）が内容をレビューし、承認されると数秒〜数分以内にGoogle Cloud環境へ自動的に権限が付与されます。
承認完了、または却下された際には、システムから申請者（あなた）宛てに**結果の通知メール**が自動送信されます。

______________________________________________________________________

## 2. フォームの入力項目と書き方

フォームには以下の項目を入力してください。必須項目が漏れている、または不正なフォーマットの場合、システムエラーとなり権限が付与されません。

### 📌 申請種別 (必須)

目的のアクションを選択します。

- **新規付与**: 新しく権限が欲しい場合。
- **変更**: 既存の期限やロールを変えたい場合。
- **削除**: 不要になった権限を消したい（返却したい）場合。
- **緊急**: システム障害対応などで、承認を待たずに今すぐ権限が必要な場合（※後述の「緊急アクセス」を参照）。

### 📌 対象プリンシパル (必須)

権限を付与・変更・削除したい対象のメールアドレスを正確に入力してください。

- ユーザー個人の場合: `your-name@example.com`
- サービスアカウントの場合: `my-sa@my-project.iam.gserviceaccount.com`
- グループの場合: `dev-team@example.com`

### 📌 対象リソース (必須)

権限の対象となるGoogle CloudのプロジェクトID、またはフォルダIDを入力してください。

- 例: `my-production-project`
- フォルダなどに付与する場合は `folders/123456789` のように完全なプレフィックスを含めてください。

### 📌 付与・変更ロール (必須)

Google Cloudの正式なIAMロール名を入力してください。

- 例: `roles/bigquery.dataViewer` (BigQueryのデータ閲覧者)
- 例: `roles/storage.objectAdmin` (Cloud Storageのオブジェクト管理者)

> **💡 どのロールを選べばいいか分からない場合**
> フォームの説明欄に記載されている\*\*「Geminiロール提案アシスタント」**のURLを開いてください。
> 「BigQueryで特定のデータセットだけを見たい」といった**やりたいこと（自然言語）\*\*を入力すると、AIがセキュリティベストプラクティスに基づいた「最小権限のロール名」を提案してくれます。

### 📌 申請理由・利用目的 (必須)

なぜこの権限が必要なのか、承認者が判断できる具体的な理由を記述してください。

- 良い例: 「〇〇プロジェクトのログ調査のため、2月末まで閲覧権限が必要」
- 悪い例: 「開発で使うため」（※理由が不明確な場合、承認者に却下される可能性があります）

### 📌 承認者メール（または承認グループ） (必須)

この申請をレビューし、承認する権限を持つ人（上長やプロジェクトマネージャーなど）のメールアドレスを入力してください。

### 📌 申請者メール (必須)

申請を行うあなた自身のメールアドレスを入力してください。このアドレス宛に、申請の「承認」または「却下」の結果通知メールが届きます。

### 📌 現在のロール付与状況について（変更・削除申請の場合のみ）

変更や削除を行いたい権限の現在の状態や、元々付与された背景などを簡潔に記載してください。承認者が変更・削除の妥当性を確認するための参考情報となります。（新規付与申請の場合は「なし」等で構いません）

### 📌 付与期間（新規・変更申請の場合）

権限が必要な期間のタイプを選択してください。

- **期間限定**: 一定期間のみ権限が必要な場合。セキュリティの観点から、原則としてこちらを利用してください。
- **恒久**: 期限を設けず、継続的に権限が必要な場合。（※本番環境の強い権限などでは、承認されない場合があります）

### 📌 付与終了日時（期間限定を選択した場合）

「期間限定」を選択した場合は、権限が不要になる日付（有効期限）を指定してください。
指定した日付の23:59に、システムが自動的に権限を剥奪（回収）するため、手動での削除申請は不要です。

## 3. 🚨 緊急時の対応（Break-glass フロー）

本番環境のシステムダウンや重大な障害発生時など、**「人間の承認を待っている時間がない」** という一刻を争う事態のための特別なフローです。

**使い方:**
申請種別で\*\*「緊急」**または**「緊急付与」\*\*を選択して申請を送信してください。

**どのような動作になるか:**

1. 人間の承認プロセスを完全にスキップし、システムが即座に（数秒以内に）権限を付与します。
1. 同時に、**「緊急アクセスが発動された」という強烈な警告アラートが、監査部門および全システム管理者に即時通知**されます。
1. 監査証跡（ログ）には「緊急承認」として永久に記録されます。

> **⚠️ 注意:**
> 緊急アクセスは障害復旧のための「最後の手段（非常ボタン）」です。不適切な利用は重大なコンプライアンス違反となります。利用後は必ず事後報告を行い、不要になった権限は速やかに「削除」申請を出してください。

## 4. よくある質問 (FAQ)

**Q. 申請したのに権限が使えません。**
A. 以下の理由が考えられます。

1. **まだ承認されていない:** 承認者に連絡し、レビューをお願いしてください。
1. **クラウド側の反映遅延:** 承認されてシステムが付与を実行した後、Google Cloud側で設定が完全に浸透するまで数分程度のタイムラグ（遅延）が発生する場合があります。5分ほど待ってから再度試してください。
1. **入力間違い:** プリンシパル（メールアドレス）やプロジェクトIDにタイポがないか確認してください。

**Q. 申請が却下された場合、どうすればよいですか？**
A. 却下されるとシステムから通知メールが届きます。メール本文に記載されている「再申請用リンク」をクリックすると、前回入力した内容が最初から埋まった状態のフォームが開きます。指摘された箇所（理由など）だけを修正して、簡単に再提出を行うことができます。

**Q. 間違った内容で申請してしまいました。**
A. すでに送信した申請は直接変更できません。承認者に連絡して「間違えたので却下してください」とお伝えください。却下されると再申請用の便利なリンク付きメールが届きますので、そこから内容を修正して再提出してください。
