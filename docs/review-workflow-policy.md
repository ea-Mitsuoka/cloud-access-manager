# IAM権限申請レビュー・ワークフロー設計方針

本ドキュメントは、Cloud Access Manager における申請のレビュー処理、フィードバック、およびUI（スプレッドシート）上のデータ保持に関する基本方針と仕様を定義する。

## 1. 基本思想 (Inbox Zero と 心理的安全性)

- **Inbox Zero**: `requests_review` シートは「未処理のタスクリスト」として機能させ、処理が完了した行は即座に画面から取り除く。
- **心理的安全性**: 処理結果をダイアログで明示し、さらに別シートへ履歴として移動させることで、承認者に「正しく処理された」という確証と振り返りの手段を提供する。
- **Single Source of Truth**: 真の監査ログは常にBigQueryに保存される。スプレッドシートのデータは一時的なキャッシュ（人間のためのUI）と割り切る。

## 2. ステータス別のシステム挙動定義

すべての正常処理において、承認者がダイアログの「OK」を押した瞬間に、該当行は `requests_review` シートから削除され、`processed_history` シートへ転記される。

### ① 承認済 (APPROVED)

- **意味**: 申請内容を妥当と判断し、GCP上に実際の権限を付与する。
- **挙動**:
  1. ステータスを「承認済」に変更し実行。
  1. GAS -> Cloud Run -> IAM API経由で権限付与。同時にBigQueryへ履歴書込。
  1. 成功レスポンス後、「✅ IAM権限の付与が完了しました」ダイアログを表示。
  1. OK押下でシート間転記（行移動）。

### ② 却下 (REJECTED)

- **意味**: ポリシー違反等により、権限付与を拒否する。
- **挙動**:
  1. ステータスを「却下」に変更し実行。
  1. **【UI要件】** GASで「却下理由」を入力するポップアップを表示（入力必須化）。
  1. GAS -> Cloud Run経由でBigQueryに却下履歴と理由を保存（IAM付与はスキップ）。
  1. バックエンドから申請者へ「却下通知（理由記載）」を自動送信。
  1. 成功レスポンス後、「🚫 申請を却下として記録しました」ダイアログを表示。
  1. OK押下でシート間転記（行移動）。

### ③ 取消 (CANCELLED)

- **意味**: 申請の取り下げ、起票ミス・重複による無効化。
- **挙動**:
  1. ステータスを「取消」に変更し実行。
  1. GAS -> Cloud Run経由でBigQueryに取消履歴を保存（IAM付与はスキップ）。
  1. 成功レスポンス後、「🗑️ 申請を取り消しました」ダイアログを表示。
  1. OK押下でシート間転記（行移動）。

### ⚠️ 例外処理 (ERROR)

- 万が一、Cloud Runとの通信エラーやIAM付与エラーが発生した場合は、**シートの行移動・削除を絶対に行わない**。
- 「❌ エラーが発生しました: [エラー内容]」ダイアログを表示し、行を維持。これにより承認者の調査・再実行を可能とする。

## 3. スプレッドシート履歴の自動ローテーション機能

スプレッドシートの肥大化とパフォーマンス低下を防ぐため、以下の自動パージ（削除）ルールを運用する。

- **対象**: `processed_history`（直近の処理済ログ）シート
- **仕組み**: GASの時間主導型トリガーを使用（毎日深夜バッチ実行）。
- **ルール**: 処理日時から**30日以上**経過している行をシートから自動削除する。
- **担保**: スプレッドシートから消えても、BigQueryには永久保存されているため監査上の問題は発生しない。

## 4. 申請者側のUX向上要件 (SaaS Portal)

却下・取消となった申請に対するリカバリ手段として、SaaS Portal側に以下の機能を実装する。

- **過去履歴タブ**: ユーザー自身の過去の申請（承認・却下・取消）を一覧表示する。
- **再申請機能**: 過去の却下/取消データに対して「コピーして再申請」ボタンを配置。以前の申請内容（リソースやロール）がプレフィルされた状態で新規起票画面を開き、データの上書きではなく新しい「申請ID」としてEvent Sourcing原則を遵守する。
