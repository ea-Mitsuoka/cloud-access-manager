# Cloud Access Manager: アーキテクチャの現状とSaaS化に向けた考察

## 1. はじめに

本ドキュメントは、Cloud Access Managerを「マルチテナント（データ分離型）SaaS」として進化させるための技術的要件と、アーキテクチャのあり方について考察したものです。現在のシステムは、セキュリティ監査の観点から、コンピュートリソース（Cloud Run）とデータストア（BigQuery）を単一の管理用プロジェクト（`tool_project_id`）に集約する設計となっています。この設計は単一組織内での運用には適していますが、外部顧客への提供を想定した場合、様々な課題が生じます。

## 2. マルチテナント化（疎結合化）に必要な3つの改修

現在の「1つのCloud Run ↔ 1つのBigQuery」という構成を、「1つのCloud Run ↔ 複数のターゲットBigQuery」という形に拡張するためには、以下の改修が必要となります。

### 2.1 Pythonバックエンドの「動的ルーティング化」

現在、Cloud Runは起動時に環境変数からBigQueryのプロジェクトIDを読み込み、固定のデータベース接続として初期化しています。これを改修し、リクエストごとに宛先のBigQueryプロジェクトを動的に切り替える仕組みが必要です。具体的には、フロントエンド（GAS）からテナントを識別する情報を受け取り、その情報に基づいて接続先のリポジトリをインスタンス化するように変更します。

### 2.2 Terraformの役割分離

現在のTerraformコードは、Cloud RunとBigQueryの構築が1つのセット（`main.tf`）になっています。これを以下の2つのモジュールに完全に分離する必要があります。

- **コントロールプレーン用Terraform:** Cloud RunやCloud Schedulerなど、SaaSベンダー側が管理する「システム本体」をデプロイします。
- **テナントプロビジョニング用Terraform:** 顧客（テナント）が増えるたびに実行し、顧客のGCPプロジェクト内にBigQueryデータセットを作成します。さらに、コントロールプレーンのCloud Run（サービスアカウント）に対して、そのBigQueryへの書き込み権限（`roles/bigquery.dataEditor` など）を付与します。

### 2.3 フロントエンドのテナント識別

顧客ごとに異なるスプレッドシート（SaaSポータル連携GAS）を配布することになります。各スプレッドシートのGASスクリプトプロパティにテナント識別情報（例: `TENANT_BQ_PROJECT_ID`）を設定し、Cloud RunへのAPIリクエスト時に、自らがどのテナントであるかをペイロードに含めて送信する改修が必要です。

## 3. データ分離型アーキテクチャのトレードオフ（顧客側BQモデル）

BigQueryを顧客側に分離する（データは顧客持ちとする）アーキテクチャには、明確なメリットとデメリットが存在します。

### 3.1 メリット

- **物理的なデータ隔離:** コンプライアンス要件に対応し、顧客間のデータ混入リスクをゼロにします。
- **課金の分離:** BigQueryのストレージやクエリ費用が各顧客に直接請求されるため、SaaS運営側のコスト按分コストを削減できます。
- **データオーナーシップの確保:** 顧客自身が自社のBigQueryに対してクエリを実行し、独自の監査や分析を行えます。

### 3.2 デメリット（運用上の課題）

- **中央監視（SRE）の困難化:** エラーログなどが各顧客のBigQueryに分散するため、システム全体を横断的に監視する難易度が跳ね上がります。
- **Cloud Run権限の肥大化とリスク:** SaaSベンダー側のCloud Runが、全顧客のIAM変更権限とBigQuery書き込み権限を併せ持つことになります。万が一侵害された場合の被害範囲（ブラストラディウス）が極めて大きくなります。
- **スキーマ変更の複雑化:** 将来的なバージョンアップでBigQueryのスキーマを変更する際、全顧客の環境に対して個別にマイグレーション処理（`ALTER TABLE`など）を実行する必要があります。

## 4. SaaSベンダー側でデータを保持する場合のテナント分離アーキテクチャ

前述のBYOC（顧客側データ保持）ではなく、SaaSベンダー側のGCP環境内に全テナントのBigQueryデータを集約してサービスを展開する（SaaS Managed Dataモデル）場合、「テナントごとにGCPプロジェクトを分けるべきか」、それとも「同一GCPプロジェクト内でBigQueryのデータセットだけを分けるべきか」という設計の分岐点が生じます。

### 4.1 データセット分離モデル（単一GCPプロジェクト）

SaaSベンダーの1つのGCPプロジェクト内に、テナントごとに `tenant_A_dataset`、`tenant_B_dataset` のようにBigQueryデータセットを作成して分離する方式です。

- **メリット:**
  - **プロビジョニングの高速化:** 新規顧客の追加がデータセット作成のみで完結するため、オンボーディングが非常に高速です。
  - **インフラ管理の集約:** Terraform等のIaCコードベースがシンプルに保てます。
- **デメリット:**
  - **ノイジー・ネイバー（Noisy Neighbor）問題:** BigQueryのAPIクォータ（制限）はGCPプロジェクト単位で適用されます。特定のテナントが重いクエリ（棚卸しマトリクスの生成など）を大量に発行してAPI制限に達すると、他の全テナントのバッチ処理や申請フローも巻き込まれて停止するリスクがあります。
  - **セキュリティ境界の脆弱さ:** データセットレベルでのIAM制御は可能ですが、プロジェクトレベルの閲覧権限（Viewer等）を運用者が誤って付与した場合、即座に全テナントのアクセスログ漏洩（クロステナント・アクセス）に直結します。

### 4.2 GCPプロジェクト分離モデル（テナント・パー・プロジェクト）

SaaSベンダーのGCP組織（フォルダ）配下に、テナントごとに専用のGCPプロジェクト（例: `saas-tenant-a-prod`）を都度作成し、その中にBigQueryをデプロイする方式です。

- **メリット:**
  - **強力なリソースとIAMの隔離:** プロジェクトが物理的に分かれているため、テナント間のデータ混入や権限の誤付与リスクがGCPのアーキテクチャレベルで厳格に遮断されます。
  - **クォータと課金の完全分離:** APIのレートリミットがプロジェクトごとに独立するため、他テナントの影響を一切受けません。また、GCPの標準機能だけでテナントごとの正確な原価計算（コストトラッキング）が可能です。
- **デメリット:**
  - **運用オーバーヘッドの増大:** 顧客が増えるたびにGCPプロジェクトの新規作成、APIの有効化、初期設定などが発生するため、Terraformを用いた高度に自動化されたテナントプロビジョニング・パイプラインの構築が必須となります。

### 4.3 考察結果：セキュリティ製品としての最適解

Cloud Access Managerのように、顧客の「IAM権限の履歴」や「アクセスログ」といった**極めて機密性の高いセキュリティ・メタデータを預かるエンタープライズSaaS**においては、**「GCPプロジェクト分離モデル（テナント・パー・プロジェクト）」を採用することが強く推奨されます**。

データセット分離モデルは開発・初期運用コストが低いものの、APIクォータの枯渇リスクや、万が一のIAM設定ミスによる他社データ漏洩のリスクが許容できないため、ゼロトラストを前提とするセキュリティ製品のアーキテクチャとしては不適格です。

## 5. 本質的な課題：ログリークと機密性のジレンマ

BigQueryを顧客側に分離、あるいはSaaS側でプロジェクト分離したとしても、コンピュート層（Cloud Run）をSaaSベンダーが管理する以上、\*\*IAMの変更履歴などの機密性の高いメタデータはSaaSベンダー側に漏洩する（筒抜けになる）\*\*という致命的な問題が残ります。

Google Cloud Runは、標準出力（`stdout` / `stderr`）に吐き出されたすべての情報を、Cloud Runが存在するプロジェクトのCloud Loggingに自動的に記録します。現在の実装では、緊急アクセスのログや例外時のトレースバックなどに、リクエストのペイロード（誰が、どのリソースに、何の権限を得たか）が平文で含まれています。結果として、SaaSベンダーの管理者はこれらのセキュリティ・メタデータをCloud Loggingから閲覧可能となってしまいます。

## 6. エンタープライズSaaSにおける3つの解決策

この「ログリーク問題」を解決し、機密性を担保するためには、以下のいずれかのアプローチを選択する必要があります。

1. **BYOC (Bring Your Own Cloud) モデル:** 実行環境を提供するのではなく、Terraformコード一式を提供し、顧客自身のGCP環境内にCloud RunもBigQueryもすべてデプロイしてもらう方式です。データもログも顧客環境から出ないため、機密性は100%（ゼロトラスト）担保されます。現在のCloud Access Managerは、実質的にこのモデル（プライベートSaaS/アプライアンス）として高い完成度を持っています。
1. **ペイロードの完全暗号化 / ログマスキング モデル:** SaaSベンダー側でマルチテナント構成を維持しつつ、Cloud Loggingへの出力を極小化し、重要なデータは顧客のKMSキーで暗号化してから保存するといった、ゼロ知識証明に近い高度な実装を行う方式です。開発難易度は非常に高くなります。
1. **「SaaSベンダーを信頼する」モデル:** 一般的なSaaS企業が採用する方式で、秘密保持契約（NDA）やSOC2などの監査レポートをベースにした法的・運用的な信頼関係を前提とします（メタデータはSaaS側のログに残るが、悪用しないという前提）。

## 7. 結論：システムの目指すべき方向性

もし本システムを「社内の複数部門」や「グループ会社」向けに大規模展開するのであれば、親会社がログを閲覧しても問題ないため、現状の「中央集約型」のまま運用する（必要に応じてRow Level Securityなどで論理的に分割する）アプローチが、運用保守の観点からは最も拡張性が高く現実的です。

一方で、「外部顧客」への外販を想定するのであれば、現在の\*\*「Terraformで顧客のGCP環境にシステム一式をプロビジョニングする（BYOCモデル）」\*\*が、セキュリティと機密性の観点から見て最もパーフェクトで訴求力のあるアーキテクチャと言えます。

## 8. 補足：分散システムにおけるバグ管理の難しさ

本システムのような、フロントエンド（GAS）、バックエンド（Cloud Run）、データストア（BigQuery）、インフラ（Terraform）が密接に連携するモダンな分散システムでは、バグの原因究明が極めて困難になります。

例えば、Pythonコードの修正が反映されない原因が、実はTerraformのデプロイ仕様（`:latest`タグの扱い）にあるといったように、インフラとアプリの境界線上にバグが潜むケースが多々あります。また、非同期処理による状態の不一致や、「良かれと思ったセキュリティ修正」がBigQueryの型制約と衝突して新たなバグを生む（デグレード）など、複雑な要因が絡み合うため、慎重なアーキテクチャ設計とテストが不可欠です。
